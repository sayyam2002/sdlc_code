# FUNCTIONAL REQUIREMENTS DOCUMENT (FRD)

## 1. Document Overview

### 1.1 Purpose
This Functional Requirements Document (FRD) translates the business requirements specified in the Business Requirement Document (BRD) for Use Case 1 into detailed, implementation-ready functional requirements for the AWS-based data processing pipeline. This document serves as the foundation for technical design and development activities.

### 1.2 Scope
This FRD covers the following functional areas:
- Ingestion of customer and order data from AWS S3
- Data cleaning and deduplication processes
- AWS Glue Catalog table registration
- SCD Type 2 implementation using Apache Hudi for order summary data
- Incremental data processing logic
- Customer aggregate spend calculation and storage
- AWS service integration (S3, Glue, Athena, CloudWatch, Lake Formation)

### 1.3 Source Document Reference
- **Source BRD**: Business Requirement Document – AWS Version
- **S3 Location**: s3://adif-sdlc/brd_usecase1/BusinessRequirementDocument_usecase1.pdf
- **Use Case**: Use Case 1

### 1.4 Assumptions
- AWS infrastructure and required services (S3, Glue, Athena, CloudWatch) are provisioned and accessible
- Appropriate IAM roles and permissions are configured for service-to-service communication
- Source data files are deposited in the specified S3 locations in CSV format
- Apache Hudi framework is available within the AWS Glue environment
- Not specified in BRD: Data arrival frequency, file naming conventions, data volume expectations, retention policies

### **FR-SCD2-001: Implement SCD Type 2 for Order Summary**
- **BRD Reference**: Section 5 (SCD Type 2), Section 6 (Incremental Logic)
- **Description**: The system shall implement Slowly Changing Dimension Type 2 logic for the ordersummary table using Apache Hudi on S3, maintaining historical records with active/inactive status and temporal attributes.
- **Inputs**: 
  - Cleaned customer dataset: CustId, Name, EmailId, Region
  - Cleaned order dataset: OrderId, ItemName, PricePerUnit, Qty, Date, CustId
  - Existing ordersummary table (for incremental runs)
- **Processing Rules**:
  - Join customer and order datasets on CustId
  - Detect changes in customer data that trigger updates
  - For new records: Insert with IsActive flag set to active, current StartDate, null EndDate, current OpTs
  - For changed records: 
    - Close previous row by setting IsActive to inactive, setting EndDate to current timestamp, updating OpTs
    - Insert new active row with updated data, IsActive set to active, current StartDate, null EndDate, current OpTs
  - Use Apache Hudi framework for upsert operations
  - Write data to S3 location: s3://adif-sdlc/curated/sdlc_wizard/ordersummary/
  - Maintain fields: IsActive, StartDate, EndDate, OpTs (in addition to joined data fields)
- **Outputs**: 
  - Hudi-managed ordersummary table at s3://adif-sdlc/curated/sdlc_wizard/ordersummary/
  - Records with SCD Type 2 attributes: IsActive, StartDate, EndDate, OpTs
- **Preconditions**: 
  - Customer and order data have been cleaned and cataloged (FR-CLEAN-001 through FR-CATALOG-002)
  - S3 location s3://adif-sdlc/curated/sdlc_wizard/ordersummary/ is accessible
  - Apache Hudi is configured in Glue environment
- **Postconditions**: 
  - Historical changes are preserved with temporal tracking
  - Only one active record exists per logical entity at any point in time
  - Previous versions are marked inactive with appropriate end dates
- **Error Conditions**:
  - Join failure due to missing CustId references
  - Hudi write operation failure
  - S3 write permission issues
  - Not specified in BRD: Change detection logic details, primary key definition

### **FR-AGG-001: Calculate and Store Customer Aggregate Spend**
- **BRD Reference**: Section 7 (Aggregation Table)
- **Description**: The system shall calculate total spending per customer and store the results in the customeraggregatespend table.
- **Inputs**: 
  - Order data with fields: OrderId, ItemName, PricePerUnit, Qty, Date, CustId
  - Customer data with fields: CustId, Name, EmailId, Region
- **Processing Rules**:
  - Join order and customer data on CustId
  - Calculate TotalAmount per customer (aggregation logic: Not specified in BRD - assumed as sum of PricePerUnit * Qty)
  - Group by customer Name and Date
  - Output columns: Name, TotalAmount, Date
  - Write results to S3 location: s3://adif-sdlc/analytics/customeraggregatespend/
- **Outputs**: 
  - customeraggregatespend table at s3://adif-sdlc/analytics/customeraggregatespend/
  - Schema: Name, TotalAmount, Date
- **Preconditions**: 
  - Customer and order data have been cleaned and cataloged (FR-CLEAN-001 through FR-CATALOG-002)
  - S3 location s3://adif-sdlc/analytics/customeraggregatespend/ is accessible
- **Postconditions**: Customer aggregate spend data is available for analytics and querying
- **Error Conditions**:
  - Aggregation calculation failure
  - S3 write permission issues
  - Join failure due to missing CustId references
  - Not specified in BRD: Handling of customers with no orders, date granularity for aggregation
