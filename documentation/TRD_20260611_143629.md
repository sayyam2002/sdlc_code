# TECHNICAL REQUIREMENTS DOCUMENT (TRD)

## 1. Document Overview

### 1.1 Purpose
This Technical Requirements Document (TRD) translates the Functional Requirements Document (FRD) for Use Case 1 into detailed, implementation-ready technical specifications for AWS Glue-based data processing pipeline. This document provides the technical blueprint for ingesting customer and order data from S3, performing data cleaning and deduplication, implementing SCD Type 2 logic using Apache Hudi, and generating aggregated analytics outputs.

### 1.2 Scope
This TRD covers technical implementation details for:
- AWS Glue ETL jobs for data ingestion from S3 CSV sources
- Data quality transformations (null removal, deduplication)
- AWS Glue Data Catalog table registration and metadata management
- SCD Type 2 implementation using Apache Hudi format on S3
- Incremental processing logic for customer data changes
- Aggregation logic for customer spending analytics
- AWS service integration: S3, AWS Glue (Spark + Hudi), Glue Data Catalog, Athena, CloudWatch
- Optional: Lake Formation governance integration

**Out of Scope:** 
- Infrastructure provisioning and IAM configuration
- Network and security architecture
- Data sources, transformations, or business rules not specified in the FRD
- Performance tuning parameters not specified in FRD
- Scheduling and orchestration mechanisms

### **TR-CLEAN-001: Implement Null Value Removal for Customer Dataset**
- **FR Reference:** FR-CLEAN-001
- **Objective:** Remove all records containing string "Null" or actual NULL values from customer dataset
- **Source Details:**
  - Source: In-memory customer_raw_df from TR-INGEST-001
  - Schema: CustId, Name, EmailId, Region
- **Target Details:** In-memory Spark DataFrame (customer_null_cleaned_df)
- **Schema:** Same as source (CustId, Name, EmailId, Region)
- **Transformation Logic:**
  - Filter out rows where any column contains actual NULL value
  - Filter out rows where any column contains string value "Null" (case-sensitive match)
  - Apply filter conditions:
    ```
    WHERE CustId IS NOT NULL 
      AND Name IS NOT NULL 
      AND EmailId IS NOT NULL 
      AND Region IS NOT NULL
      AND CustId != 'Null'
      AND Name != 'Null'
      AND EmailId != 'Null'
      AND Region != 'Null'
    ```
  - Retain all other records unchanged
- **Validation Rules:**
  - Verify output DataFrame contains no NULL values in any column
  - Verify output DataFrame contains no "Null" string values in any column
  - Log count of records removed
  - Validate at least one record remains after filtering (warn if zero records)
- **Error Handling:**
  - If all records are removed (entire dataset is null): Log warning "All customer records removed during null cleaning - zero records remaining" and continue (not specified as failure condition in FRD)
  - If input DataFrame is empty: Log warning "Input customer DataFrame is empty" and return empty DataFrame
- **Runtime Parameters:**
  - null_string_value: "Null"
- **Operational Notes:**
  - This transformation is applied after ingestion (TR-INGEST-001) and before deduplication (TR-CLEAN-003)
  - Both actual NULL and string "Null" must be removed as per FRD specification
  - Output DataFrame will be input to TR-CLEAN-003

### **TR-SCD2-001: Implement SCD Type 2 Logic for OrderSummary Table Using Hudi**
- **FR Reference:** FR-SCD2-001
- **Objective:** Implement Slowly Changing Dimension Type 2 logic for ordersummary table using Apache Hudi format to maintain historical records with versioning metadata
- **Source Details:**
  - Source 1: In-memory customer_cleaned_df from TR-CLEAN-003
    - Schema: CustId, Name, EmailId, Region
  - Source 2: In-memory order_cleaned_df from TR-CLEAN-004
    - Schema: OrderId, ItemName, PricePerUnit, Qty, Date, CustId
  - Source 3: Existing ordersummary Hudi table (if present)
    - Location: `s3://adif-sdlc/curated/sdlc_wizard/ordersummary/`
    - Format: Hudi
- **Target Details:**
  - Target Table: ordersummary
  - Storage Location: `s3://adif-sdlc/curated/sdlc_wizard/ordersummary/`
  - Storage Format: Apache Hudi
  - Schema: Not fully specified in FRD (derived from join of customer + order, plus SCD2 metadata columns)
  - Expected columns: 
    - From Customer: CustId, Name, EmailId, Region
    - From Order: OrderId, ItemName, PricePerUnit, Qty, Date
    - SCD2 Metadata: IsActive, StartDate, EndDate, OpTs
- **Schema:** 
  ```
  CustId: (type from source)
  Name: (type from source)
  EmailId: (type from source)
  Region: (type from source)
  OrderId: (type from source)
  ItemName: (type from source)
  PricePerUnit: (type from source)
  Qty: (type from source)
  Date: (type from source)
  IsActive: Boolean (SCD2 metadata)
  StartDate: Timestamp (SCD2 metadata)
  EndDate: Timestamp (SCD2 metadata)
  OpTs: Timestamp (SCD2 metadata - operation timestamp)
  ```
- **Transformation Logic:**
  - **Step 1: Join Customer and Order Data**
    - Perform join between customer_cleaned_df and order_cleaned_df on CustId
    - Join type: Not specified in FRD (assume inner join to match orders with valid customers)
    - Result: joined_df with all columns from both sources
  
  - **Step 2: Read Existing Hudi Table (if exists)**
    - Attempt to read existing ordersummary Hudi table from S3
    - If table does not exist, treat as initial load
    - If table exists, read active records (IsActive = true)
  
  - **Step 3: Detect Changes**
    - Compare incoming joined_df records with existing active records
    - Change detection key: Not specified in FRD (assume CustId + OrderId as composite key)
    - Identify: New records, Changed records, Unchanged records
    - Change detection logic: Not fully specified in FRD
  
  - **Step 4: Apply SCD Type 2 Logic**
    - For changed records:
      - Close previous rows: Set IsActive = false, update EndDate = current_timestamp
      - Insert new rows: Set IsActive = true, StartDate = current_timestamp, EndDate = null
    - For new records:
      - Insert with IsActive = true, StartDate = current_timestamp, EndDate = null
    - For unchanged records:
      - No action (keep existing active records)
    - Set OpTs = current_timestamp for all new/updated records
  
  - **Step 5: Write to Hudi Table**
    - Use Hudi upsert operation
    - Write mode: Upsert (update existing, insert new)
    - Hudi table type: Not specified in FRD (assume Copy-on-Write for SCD2)
    - Hudi record key: Not specified in FRD (assume CustId + OrderId)
    - Hudi precombine field: OpTs (for conflict resolution)
    - Hudi partition field: Not specified in FRD
- **Validation Rules:**
  - Verify join between customer and order produces expected records
  - Verify all active records have IsActive = true and EndDate = null
  - Verify all closed records have IsActive = false and EndDate is not null
  - Verify StartDate <= EndDate for all closed records
  - Verify OpTs is populated for all records
  - Validate Hudi write completes successfully
  - Verify data is readable from Hudi table after write
- **Error Handling:**
  - If join fails due to data type mismatch: Log error "Join failed between customer and order - data type mismatch on CustId" and fail job
  - If Hudi write fails: Log error "Hudi write operation failed for ordersummary table" with exception details and fail job
  - If S3 write permission issues: Log error "Insufficient permissions to write to s3://adif-sdlc/curated/sdlc_wizard/ordersummary/" and fail job
  - If Hudi table read fails (for existing table): Log error "Failed to read existing Hudi table" and fail job
- **Runtime Parameters:**
  - target_hudi_path: "s3://adif-sdlc/curated/sdlc_wizard/ordersummary/"
  - hudi_table_name: "ordersummary"
  - hudi_table_type: Not specified in FRD (recommend "COPY_ON_WRITE")
  - hudi_record_key: Not specified in FRD (recommend "CustId,OrderId")
  - hudi_precombine_field: "OpTs"
  - hudi_partition_field: Not specified in FRD
  - join_type: Not specified in FRD (recommend "inner")
  - change_detection_columns: Not specified in FRD
- **Operational Notes:**
  - This is a complex transformation combining join, change detection, and SCD2 logic
  - FRD does not specify exact join type, change detection key columns, or specific logic for determining "change"
  - Hudi configuration parameters (record key, partition field, table type) not fully specified in FRD
  - Implementation requires decisions on: join type, change detection strategy, Hudi configuration
