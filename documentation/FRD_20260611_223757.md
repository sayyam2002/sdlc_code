# FUNCTIONAL REQUIREMENTS DOCUMENT (FRD)

## 1. Document Overview

### 1.1 Purpose
This Functional Requirements Document (FRD) translates the business requirements specified in the BRD for Use Case 1 into detailed, implementation-ready functional requirements for the AWS-based data processing pipeline. The system will ingest customer and order data from S3, perform data cleaning, implement SCD Type 2 logic using Hudi, and generate aggregated analytics outputs.

### 1.2 Scope
This FRD covers:
- Ingestion of customer and order CSV files from AWS S3
- Data cleaning and deduplication processes
- AWS Glue Catalog table registration
- SCD Type 2 implementation for order summary using Apache Hudi
- Incremental data processing logic
- Customer aggregate spend calculation
- AWS architecture components for ETL, storage, cataloging, querying, and monitoring

### 1.3 Source Document Reference
- **Source BRD**: Business Requirement Document – AWS Version
- **S3 Path**: s3://adif-sdlc/brd_usecase1/BusinessRequirementDocument_usecase1.pdf

### 1.4 Assumptions
- AWS infrastructure (S3, Glue, Athena, CloudWatch) is provisioned and accessible
- Source CSV files follow the specified schemas consistently
- Lake Formation configuration is optional and not mandatory for initial implementation
- Date fields in source data are in a valid, parseable format
- CustId serves as the join key between Customer and Order datasets

### FR-INGEST-001: Ingest Customer Data from S3
- **BRD Reference**: Section 1 (Source Input)
- **Description**: The system shall read customer data in CSV format from the specified S3 location.
- **Inputs**: 
  - Source Path: s3://adif-sdlc/sdlc_wizard/customerdata/
  - Schema: CustId, Name, EmailId, Region
- **Processing Rules**:
  - Read all CSV files from the specified S3 path
  - Parse CSV with header row containing: CustId, Name, EmailId, Region
- **Outputs**: Raw customer dataset with fields: CustId, Name, EmailId, Region
- **Preconditions**: 
  - S3 bucket and path must be accessible
  - CSV files must exist in the source location
- **Postconditions**: Customer data loaded into processing memory/dataframe
- **Error Conditions**:
  - S3 path not accessible or does not exist
  - CSV files malformed or missing required columns
  - Insufficient permissions to read from S3

### FR-SCD2-001: Implement SCD Type 2 for Order Summary Table
- **BRD Reference**: Section 5 (SCD Type 2) and Section 6 (Incremental Logic)
- **Description**: The system shall implement Slowly Changing Dimension Type 2 logic for the ordersummary table using Apache Hudi on S3, maintaining historical records with active/inactive status.
- **Inputs**: 
  - Cleaned customer dataset (CustId, Name, EmailId, Region)
  - Cleaned order dataset (OrderId, ItemName, PricePerUnit, Qty, Date, CustId)
  - Existing ordersummary table (if present)
- **Processing Rules**:
  - Join customer and order datasets on CustId
  - Compare incoming records with existing ordersummary records
  - On data change detection:
    - Insert new records with IsActive flag set to active
    - Update previous records by setting IsActive to inactive and populating EndDate
  - Maintain SCD Type 2 fields: IsActive, StartDate, EndDate, OpTs
  - Use Apache Hudi format for storage
- **Outputs**: 
  - Target Table: ordersummary
  - Location: s3://adif-sdlc/curated/sdlc_wizard/ordersummary/
  - Format: Apache Hudi
  - Additional Fields: IsActive, StartDate, EndDate, OpTs
- **Preconditions**: 
  - Customer and order data have been cleaned and deduplicated
  - S3 target location is accessible
  - Hudi libraries are available in Glue environment
- **Postconditions**: 
  - ordersummary table contains current and historical records
  - Only one active record exists per unique business key
  - Historical records are preserved with appropriate date ranges
- **Error Conditions**:
  - Join failure due to missing CustId
  - S3 write permission issues
  - Hudi format write failures

### FR-SCD2-002: Process Incremental Customer Updates
- **BRD Reference**: Section 6 (Incremental Logic)
- **Description**: The system shall detect customer data updates and trigger corresponding SCD Type 2 updates in the ordersummary table.
- **Inputs**: 
  - New/updated customer records (CustId, Name, EmailId, Region)
  - Existing ordersummary table
- **Processing Rules**:
  - Identify customer records that have changed since last processing
  - Trigger SCD Type 2 update logic for affected records in ordersummary
  - Maintain IsActive, StartDate, EndDate, OpTs fields during updates
- **Outputs**: Updated ordersummary table with new active records and closed historical records
- **Preconditions**: 
  - Initial ordersummary table exists
  - Customer change detection mechanism is in place
- **Postconditions**: ordersummary table reflects latest customer information with historical tracking
- **Error Conditions**:
  - Change detection failure
  - SCD Type 2 update logic failure
