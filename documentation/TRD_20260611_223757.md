# TECHNICAL REQUIREMENTS DOCUMENT (TRD)

## 1. Document Overview

### 1.1 Purpose
This Technical Requirements Document (TRD) translates the Functional Requirements Document (FRD) into detailed technical specifications for implementing an AWS Glue-based data processing pipeline. The pipeline will ingest customer and order data from S3, perform data cleaning and deduplication, implement SCD Type 2 logic using Apache Hudi, and generate aggregated analytics outputs. This document provides implementation-ready technical specifications for AWS Glue job development.

### 1.2 Scope
This TRD covers the technical implementation of:
- AWS Glue job specifications for ingesting CSV files from S3
- Data cleaning and deduplication transformation logic using PySpark
- AWS Glue Data Catalog table registration procedures
- SCD Type 2 implementation using Apache Hudi format
- Incremental data processing patterns
- Customer aggregate spend calculation logic
- Error handling and validation rules
- Runtime parameters and operational configurations

### TR-INGEST-001: Implement Customer Data Ingestion from S3
- **FR Reference**: FR-INGEST-001
- **Objective**: Read customer CSV files from S3 into a Spark DataFrame for downstream processing
- **Source Details**: 
  - Source Type: CSV files
  - Source Location: s3://adif-sdlc/sdlc_wizard/customerdata/
  - Source Schema: CustId, Name, EmailId, Region
  - File Format: CSV with header row
- **Target Details**: In-memory Spark DataFrame (intermediate processing layer)
- **Schema**:
  ```
  CustId: string (inferred from CSV)
  Name: string (inferred from CSV)
  EmailId: string (inferred from CSV)
  Region: string (inferred from CSV)
  ```
- **Transformation Logic**:
  - Use Spark CSV reader with header=true option
  - Read all CSV files matching pattern s3://adif-sdlc/sdlc_wizard/customerdata/*.csv
  - Infer schema from CSV header row
  - Load complete dataset into DataFrame
- **Validation Rules**:
  - Verify S3 path exists and is accessible
  - Validate presence of required columns: CustId, Name, EmailId, Region
  - Confirm at least one CSV file exists in source location
  - Validate CSV files contain header row
- **Error Handling**:
  - If S3 path does not exist: Raise exception with message "Source path s3://adif-sdlc/sdlc_wizard/customerdata/ not found"
  - If no CSV files found: Raise exception with message "No CSV files found in source location"
  - If required columns missing: Raise exception with message "Required columns missing: [list missing columns]"
  - If S3 access denied: Raise exception with message "Insufficient permissions to read from S3 path"
  - Log all errors to CloudWatch with ERROR severity
- **Runtime Parameters**:
  - `--customer_source_path`: s3://adif-sdlc/sdlc_wizard/customerdata/
  - `--csv_header`: true
  - `--csv_inferSchema`: true
- **Operational Notes**:

### TR-CLEAN-001: Implement Null Value Removal for Customer Dataset
- **FR Reference**: FR-CLEAN-001
- **Objective**: Remove records containing string "Null" or NULL values from customer dataset
- **Source Details**: 
  - Source: In-memory customer DataFrame from TR-INGEST-001
  - Schema: CustId, Name, EmailId, Region
- **Target Details**: Cleaned in-memory customer DataFrame
- **Schema**: Same as source (CustId, Name, EmailId, Region)
- **Transformation Logic**:
  - Apply filter to remove rows where any column contains string literal "Null" (case-sensitive)
  - Apply filter to remove rows where any column is NULL (null value)
  - Use PySpark filter() or where() with conditions:
    ```
    Filter out rows where:
    - CustId == "Null" OR CustId IS NULL
    - Name == "Null" OR Name IS NULL
    - EmailId == "Null" OR EmailId IS NULL
    - Region == "Null" OR Region IS NULL
    ```
  - Combine conditions using OR logic
  - Negate combined condition to keep only valid rows
- **Validation Rules**:
  - Verify input DataFrame is not empty
  - Log count of records removed
  - Validate output DataFrame contains no "Null" strings or NULL values
- **Error Handling**:
  - If input DataFrame is empty: Log warning "Input customer DataFrame is empty"
  - If all records filtered out: Log warning "All customer records removed during null cleaning"
  - Log record counts before and after cleaning to CloudWatch
- **Runtime Parameters**: None
- **Operational Notes**:
  - This transformation is applied in-memory before catalog registration
  - Monitor percentage of records filtered to detect data quality issues

### TR-SCD2-001: Implement SCD Type 2 Logic for Order Summary Using Apache Hudi
- **FR Reference**: FR-SCD2-001
- **Objective**: Implement Slowly Changing Dimension Type 2 pattern for ordersummary table using Apache Hudi to maintain historical records with active/inactive status tracking
- **Source Details**: 
  - Source 1: Deduplicated customer DataFrame from TR-CLEAN-003
    - Schema: CustId, Name, EmailId, Region
  - Source 2: Deduplicated order DataFrame from TR-CLEAN-004
    - Schema: OrderId, ItemName, PricePerUnit, Qty, Date, CustId
  - Source 3: Existing ordersummary Hudi table (if present)
    - Location: s3://adif-sdlc/curated/sdlc_wizard/ordersummary/
- **Target Details**: 
  - Table Name: ordersummary
  - Storage Location: s3://adif-sdlc/curated/sdlc_wizard/ordersummary/
  - Storage Format: Apache Hudi
  - Catalog Database: gen_ai_poc_databrickscoe
  - Catalog Table: ordersummary
- **Schema**:
  ```
  Source fields from customer-order join (exact schema not fully specified in FRD):
  - CustId: string
  - Name: string
  - EmailId: string
  - Region: string
  - OrderId: string
  - ItemName: string
  - PricePerUnit: string
  - Qty: string
  - Date: string
  
  SCD Type 2 fields:
  - IsActive: boolean (true for active records, false for inactive)
  - StartDate: timestamp (record effective start date)
  - EndDate: timestamp (record effective end date, null for active records)
  - OpTs: timestamp (operation timestamp)
  ```
- **Transformation Logic**:
  1. **Join customer and order datasets**:
     - Perform inner join on CustId field
     - Result: Combined dataset with all fields from both sources
  
  2. **Read existing Hudi table** (if exists):
     - Read from s3://adif-sdlc/curated/sdlc_wizard/ordersummary/
     - Use Hudi read format with Spark
  
  3. **Identify changes**:
     - Compare incoming joined records with existing ordersummary records
     - Business key for comparison: Not specified in FRD (assume combination of CustId and OrderId)
     - Detect new records (not present in existing table)
     - Detect changed records (present but with different attribute values)
     - Detect unchanged records
  
  4. **Apply SCD Type 2 logic**:
     - For new records:
       - Insert with IsActive = true
       - Set StartDate = current timestamp
       - Set EndDate = null
       - Set OpTs = current timestamp
     
     - For changed records:
       - Update existing record: Set IsActive = false, EndDate = current timestamp
       - Insert new record: Set IsActive = true, StartDate = current timestamp, EndDate = null, OpTs = current timestamp
     
     - For unchanged records:
       - No action required
  
  5. **Write to Hudi table**:
     - Use Hudi upsert operation
     - Configure Hudi options:
       - Table type: Not specified in FRD (assume COPY_ON_WRITE)
       - Record key: Not specified in FRD (assume CustId + OrderId)
       - Precombine field: OpTs
       - Partition path: Not specified in FRD
- **Validation Rules**:
  - Verify join produces valid results (no null CustId in join output)
  - Validate only one active record exists per business key after SCD processing
  - Confirm all historical records have IsActive = false and non-null EndDate
  - Verify all active records have IsActive = true and null EndDate
  - Validate OpTs is populated for all records
- **Error Handling**:
  - If join fails due to missing CustId: Log error "Join failure: CustId missing in source data"
  - If S3 write fails: Raise exception "Failed to write to s3://adif-sdlc/curated/sdlc_wizard/ordersummary/"
  - If Hudi write fails: Log Hudi error details and raise exception
  - If multiple active records detected for same business key: Raise data quality exception
  - Log all SCD operations (inserts, updates) to CloudWatch with counts
- **Runtime Parameters**:
  - `--ordersummary_target_path`: s3://adif-sdlc/curated/sdlc_wizard/ordersummary/
  - `--hudi_table_name`: ordersummary
  - `--hudi_record_key`: Not specified in FRD
  - `--hudi_precombine_field`: OpTs
  - `--hudi_partition_path`: Not specified in FRD
  - `--hudi_table_type`: Not specified in FRD
- **Operational Notes**:
  - Hudi libraries must be available in Glue environment (use --extra-jars or marketplace connector)
  - IAM role requires s3:PutObject and s3:GetObject permissions on target path
  - Monitor Hudi commit timeline for successful writes
  - Business key definition required for proper SCD implementation (not specified in FRD)
  - Partition strategy should be defined for optimal query performance (not specified in FRD)
