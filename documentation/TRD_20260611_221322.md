## 1. Document Overview

### 1.1 Purpose
This Technical Requirements Document (TRD) translates the Functional Requirements Document (FRD) into detailed technical specifications for implementing the AWS Glue-based data processing pipeline. This document provides implementation-ready technical design for AWS Glue jobs, including data sources, transformations, catalog operations, and Apache Hudi SCD Type 2 implementation.

This TRD covers the following technical implementation areas:
- AWS Glue job specifications for data ingestion from S3
- PySpark-based data cleaning and deduplication transformations
- AWS Glue Data Catalog table registration operations
- Apache Hudi-based SCD Type 2 implementation for order summary
- Incremental data processing using Hudi upsert operations
- Customer aggregate spend calculation using Spark SQL
- AWS service integration patterns (S3, Glue, Athena, CloudWatch)
- Error handling and validation strategies
- Runtime parameters and job configurations

### **TR-SCD2-001: Implement SCD Type 2 Logic for Order Summary Using Apache Hudi**
- **FR Reference**: FR-SCD2-001
- **Objective**: Implement Slowly Changing Dimension Type 2 for ordersummary table using Apache Hudi, maintaining historical records with temporal tracking
- **Source Details**: 
  - Source 1: Cleaned customer DataFrame from TR-CLEAN-003 (CustId, Name, EmailId, Region)
  - Source 2: Cleaned order DataFrame from TR-CLEAN-004 (OrderId, ItemName, PricePerUnit, Qty, Date, CustId)
  - Source 3: Existing ordersummary Hudi table at s3://adif-sdlc/curated/sdlc_wizard/ordersummary/ (for incremental runs)
- **Target Details**: 
  - Target Type: Apache Hudi table
  - Target Path: s3://adif-sdlc/curated/sdlc_wizard/ordersummary/
  - Table Type: Copy-on-Write (COW) or Merge-on-Read (MOR) - not specified in FRD
  - Write Operation: Upsert
- **Schema**: 
  ```
  # Business fields (from join of customer and order):
  CustId: String
  Name: String
  EmailId: String
  Region: String
  OrderId: String
  ItemName: String
  PricePerUnit: Double
  Qty: Integer
  Date: String
  
  # SCD Type 2 fields:
  IsActive: Boolean (true for active records, false for inactive)
  StartDate: Timestamp (record effective start date)
  EndDate: Timestamp (record effective end date, null for active records)
  OpTs: Timestamp (operation timestamp)
  ```
  Note: Primary key definition not specified in FRD - assume CustId + OrderId or surrogate key
- **Transformation Logic**:
  1. **Join customer and order data**:
     - Join type: Inner join (not specified in FRD)
     - Join key: CustId
     - Result: Combined dataset with all fields from both sources
  
  2. **For initial load (ordersummary table does not exist)**:
     - Add SCD Type 2 columns:
       - IsActive = true
       - StartDate = current_timestamp()
       - EndDate = null
       - OpTs = current_timestamp()
     - Write to Hudi table using INSERT operation
  
  3. **For incremental load (ordersummary table exists)**:
     - Read existing ordersummary Hudi table
     - Identify new records (not in existing table)
     - Identify changed records (exists in table but data changed)
     - Change detection logic (not specified in FRD): Compare all business fields
     
     - For new records:
       - Add SCD Type 2 columns:
         - IsActive = true
         - StartDate = current_timestamp()
         - EndDate = null
         - OpTs = current_timestamp()
     
     - For changed records:
       - Close previous row:
         - Set IsActive = false
         - Set EndDate = current_timestamp()
         - Update OpTs = current_timestamp()
       - Insert new active row:
         - IsActive = true
         - StartDate = current_timestamp()
         - EndDate = null
         - OpTs = current_timestamp()
     
     - Use Hudi UPSERT operation with precombine field (OpTs)
  
  4. **Hudi configuration**:
     - Table type: Not specified in FRD (recommend COW for simplicity)
     - Record key: Not specified in FRD (assume CustId + OrderId or generate surrogate key)
     - Precombine field: OpTs
     - Partition path: Not specified in FRD
     - Hudi write operation: upsert
- **Validation Rules**:
  - Verify join produces valid results (no null keys)
  - Verify only one active record exists per logical entity (IsActive=true)
  - Verify closed records have EndDate populated
  - Verify StartDate < EndDate for closed records
  - Verify OpTs is populated for all records
  - Verify Hudi table metadata is valid
- **Error Handling**:
  - If join fails due to missing CustId: Log error with details and skip orphaned records
  - If Hudi write operation fails: Log error details and raise exception
  - If S3 write permission denied: Raise exception with IAM permission details
  - If change detection fails: Log error and raise exception
  - If multiple active records detected for same entity: Raise data quality exception
- **Runtime Parameters**:
  - hudi_table_name: ordersummary
  - hudi_table_path: s3://adif-sdlc/curated/sdlc_wizard/ordersummary/
  - hudi_table_type: Not specified in FRD (recommend COPY_ON_WRITE)
  - hudi_record_key: Not specified in FRD (requires clarification)
  - hudi_precombine_field: OpTs
  - hudi_partition_path: Not specified in FRD
  - hudi_operation: upsert
- **Operational Notes**:
  - Primary key definition not specified in FRD - critical for Hudi record key configuration
  - Change detection logic not specified in FRD - assume compare all business fields
  - Hudi table type (COW vs MOR) not specified in FRD - recommend COW for simplicity
  - Partition strategy not specified in FRD - consider partitioning by Date or Region
  - Verify IAM role has s3:PutObject permission for target path
  - Configure Hudi for Athena compatibility (set appropriate Hudi configurations)
  - This is a complex operation - consider breaking into sub-jobs for initial vs incremental load

- **FR Reference**: FR-INCR-001
- **Objective**: Process incremental customer data updates and trigger SCD Type 2 updates in ordersummary table
- **Source Details**: 
  - Source 1: New/updated customer records from s3://adif-sdlc/sdlc_wizard/customerdata/ (CustId, Name, EmailId, Region)
  - Source 2: Existing ordersummary Hudi table at s3://adif-sdlc/curated/sdlc_wizard/ordersummary/
- **Target Details**: 
  - Target: Updated ordersummary Hudi table at s3://adif-sdlc/curated/sdlc_wizard/ordersummary/
- **Schema**: Same as TR-SCD2-001 ordersummary schema
- **Transformation Logic**:
  1. **Identify changed customer records**:
     - Read new customer data from s3://adif-sdlc/sdlc_wizard/customerdata/
     - Read existing ordersummary table (active records only: IsActive=true)
     - Compare customer fields (Name, EmailId, Region) between new and existing data
     - Identify records where CustId exists but data has changed
     - Change detection mechanism not specified in FRD
  
  2. **Trigger SCD Type 2 update**:
     - For each changed customer record:
       - Retrieve all associated order records from ordersummary
       - Apply SCD Type 2 logic from TR-SCD2-001:
         - Close existing active records (IsActive=false, EndDate=current_timestamp())
         - Insert new active records with updated customer data (IsActive=true, StartDate=current_timestamp())
     - Use Hudi upsert operation with OpTs as precombine field
  
  3. **Maintain temporal integrity**:
     - Ensure IsActive flag is correctly set
     - Ensure StartDate and EndDate are properly maintained
     - Update OpTs to current timestamp for all affected records
- **Validation Rules**:
  - Verify changed customer records are correctly identified
  - Verify only affected records are updated (not full table refresh)
  - Verify SCD Type 2 integrity maintained (only one active record per entity)
  - Verify historical records are preserved
  - Log count of customer records changed and ordersummary records affected
- **Error Handling**:
  - If change detection fails: Log error and raise exception
  - If SCD Type 2 update fails: Log error details and raise exception
  - If orphaned order records found (CustId not in customer data): Log warning and handle based on business rules (not specified in FRD)
  - If Hudi upsert fails: Log error and raise exception
- **Runtime Parameters**:
  - customer_source_path: s3://adif-sdlc/sdlc_wizard/customerdata/
