# FUNCTIONAL REQUIREMENTS DOCUMENT (FRD)

## 1. Document Overview

### 1.1 Purpose
This Functional Requirements Document (FRD) translates the business requirements specified in the BRD for Use Case 1 into detailed, implementation-ready functional requirements for the AWS-based data processing pipeline. The system will ingest customer and order data from S3, perform data cleaning, implement SCD Type 2 logic using Hudi, and generate aggregated analytics outputs.

### 1.2 Scope
This FRD covers:
- Ingestion of Customer and Order CSV files from AWS S3
- Data cleaning and deduplication processes
- AWS Glue Catalog table registration
- SCD Type 2 implementation for ordersummary table using Hudi format
- Incremental processing logic triggered by customer updates
- Aggregation logic for customeraggregatespend table
- AWS architecture components: S3, AWS Glue, Glue Data Catalog, Athena, CloudWatch, and optional Lake Formation

**Out of Scope:** Any data sources, transformations, or business rules not explicitly mentioned in the source BRD.

### 1.3 Source Document Reference
- **Source BRD:** Business Requirement Document – AWS Version
- **S3 Path:** s3://adif-sdlc/brd_usecase1/BusinessRequirementDocument_usecase1.pdf
- **BRD Sections Referenced:** Sections 1-8

### 1.4 Assumptions
- AWS infrastructure (S3, Glue, Athena, CloudWatch) is provisioned and accessible
- Appropriate IAM roles and permissions are configured for Glue jobs to access S3 buckets
- CSV files in source locations follow the schemas defined in BRD Section 2
- Hudi framework is available and configured within AWS Glue Spark environment
- Lake Formation implementation is optional and not mandatory for core functionality

- **Postconditions:** Customer data is loaded into processing pipeline

- **Description:** The system shall remove all records containing "Null" (string) or NULL (actual null) values from the customer dataset.
- **Inputs:** Customer DataFrame with columns: CustId, Name, EmailId, Region

- **Preconditions:** Customer data has been ingested (FR-INGEST-001)

### **FR-SCD2-001: Implement SCD Type 2 for OrderSummary Table**
- **BRD Reference:** Section 5 (SCD Type 2), Section 6 (Incremental Logic)
- **Description:** The system shall implement Slowly Changing Dimension Type 2 logic for the ordersummary table using Apache Hudi format on S3, maintaining historical records with versioning metadata.
- **Inputs:** 
  - Cleaned Customer DataFrame (CustId, Name, EmailId, Region)
  - Cleaned Order DataFrame (OrderId, ItemName, PricePerUnit, Qty, Date, CustId)
  - Existing ordersummary table (if present)
- **Processing Rules:**
  - Join customer and order datasets on CustId
  - Compare incoming records with existing active records in ordersummary table
  - On change detection:
    - Close previous rows by setting IsActive to false (or equivalent), updating EndDate
    - Insert new rows with active status, setting StartDate
  - Maintain metadata columns: IsActive, StartDate, EndDate, OpTs
  - Write data in Hudi format to support upsert operations
  - Not specified in BRD: specific join type, change detection key columns, or exact logic for determining "change"
- **Outputs:** 
  - Target Table: ordersummary
  - Location: `s3://adif-sdlc/curated/sdlc_wizard/ordersummary/`
  - Format: Hudi
  - Columns: Not specified in BRD (derived from join of customer + order, plus IsActive, StartDate, EndDate, OpTs)
- **Preconditions:** 
  - Customer and Order data have been cleaned and cataloged
  - S3 target location is accessible
  - Hudi libraries are available in Glue environment
- **Postconditions:** 
  - ordersummary table contains current and historical records
  - Active records have IsActive = true (or equivalent)
  - Historical records have IsActive = false (or equivalent) with appropriate EndDate
- **Error Conditions:**
  - Join failure due to data type mismatch
  - Hudi write failure
  - S3 write permission issues
